# Feature Specification: AI Live Chat Support Agent

**Feature Branch**: `001-live-chat-agent`
**Created**: 2026-06-25
**Status**: Implemented
**Source**: Spur Founding Full-Stack Engineer Take-Home Assignment

## Overview

A mini AI support agent embedded in a live chat widget. Visitors open a web page, type questions about the store (shipping, returns, support hours), and receive real-time, context-aware answers generated by an LLM. Every conversation is persisted so sessions survive page reloads.

## System Context / Request Flow

```text
[Browser User]
    │
    ▼
[Next.js UI — /chat page]
    │  POST /api/chat/message  (Next.js proxy route)
    ▼
[Hono API — /chat/message]
    │
    ├──► [PostgreSQL — chat_conversations, chat_messages]
    │
    └──► [LLM API — Gemini 1.5 Flash  OR  OpenAI GPT-4o-mini]
              (10-second hard timeout via AbortController)
```

### Hot-Path Request Flow
1. User types a message and clicks **Send** (or presses Enter).
2. Frontend disables input and shows an animated typing indicator.
3. Frontend calls `POST /api/chat/message` with `{ message, sessionId? }`.
4. Next.js proxy forwards request to the Hono backend at `/chat/message`.
5. Backend validates: non-empty, ≤ 2000 characters.
6. Backend upserts a `chat_conversations` row for the session UUID.
7. Backend fetches the last 10 messages from `chat_messages` (conversation history).
8. Backend constructs prompt: system FAQ + history + new user message.
9. Backend calls LLM (Gemini preferred, OpenAI fallback) with 10 s timeout.
10. Backend saves user message AND AI reply to `chat_messages`.
11. Backend updates `chat_conversations.updated_at`.
12. Backend returns `{ reply: string, sessionId: string }`.
13. Frontend appends the reply bubble, re-enables input, auto-scrolls to bottom.

### Failure Boundaries
- **LLM timeout / API error**: Caught in the LLM service; backend returns a friendly fallback string rather than a 500 crash.
- **No API key configured**: Backend returns a demo-mode message (no crash).
- **Database error**: Returned as HTTP 500 `{ error: "DatabaseError" }`; never unhandled.
- **Empty / oversized input**: HTTP 400 with `{ error: "MessageCannotBeEmpty" }` or `{ error: "MessageTooLong" }`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Anonymous Instant Chat (Priority: P1)

Any visitor can open `/chat` and ask a question immediately — no login required.

**Why this priority**: Core value of the widget; every other story depends on this working.

**Independent Test**: Navigate to `/chat`, type "Hello", press Enter. Expect a response bubble within a few seconds.

**Acceptance Scenarios**:

1. **Given** a user visits `/chat` for the first time, **When** they type a message and press Enter, **Then** their message appears as a right-aligned blue bubble, a typing indicator animates while the AI thinks, and the AI reply appears as a left-aligned dark bubble.
2. **Given** the user has an active session, **When** they reload the page, **Then** the full conversation history is re-fetched from the backend and rendered in order.
3. **Given** the user clicks **New Chat**, **When** the action completes, **Then** the chat area clears and a fresh session ID is generated.

---

### User Story 2 — Store FAQ Answers (Priority: P2)

The AI agent answers questions about Lobstral Store policies accurately using seeded domain knowledge.

**Why this priority**: Defines the agent's purpose — wrong or hallucinated answers undermine trust.

**Independent Test**: Ask "Do you ship to the USA?" → Expect mention of 5–7 business days.

**Acceptance Scenarios**:

1. **Given** the user asks "How long does shipping take?", **When** the AI responds, **Then** the reply references domestic (5–7 business days) and international (7–14 business days) timelines.
2. **Given** the user asks "What is your return policy?", **When** the AI responds, **Then** the reply mentions the 30-day window and free domestic returns.
3. **Given** the user asks something outside the seeded policies, **When** the AI responds, **Then** it politely redirects the user to `support@lobstral-store.com` rather than guessing.

---

### User Story 3 — Contextual Follow-up Conversations (Priority: P3)

The agent maintains context across turns within the same session.

**Why this priority**: Natural conversations require memory; single-turn mode feels robotic.

**Independent Test**: Ask "What is your return policy?" then follow up "Is it free?" — expect the AI to answer in context.

**Acceptance Scenarios**:

1. **Given** the user previously asked about returns, **When** they ask "How long do I have?", **Then** the AI answers "30 days" without the user having to repeat themselves.

---

### Edge Cases

- **Empty message**: Frontend blocks submission (button disabled); backend rejects with HTTP 400 `MessageCannotBeEmpty`.
- **Message > 2000 chars**: Backend rejects with HTTP 400 `MessageTooLong`.
- **LLM API down or timed out (>10 s)**: Backend catches the AbortError/fetch error and returns a friendly fallback message; never crashes.
- **No API key in environment**: Backend returns a demo-mode message explaining the missing config.
- **Malformed JSON body**: `c.req.json().catch()` pattern returns `{}`, treated as missing fields → HTTP 400.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST expose `POST /chat/message` accepting `{ message: string, sessionId?: string }` and returning `{ reply: string, sessionId: string }`.
- **FR-002**: System MUST expose `GET /chat/history/:sessionId` returning `{ messages: Array<{ id, sender, text, timestamp }> }`.
- **FR-003**: System MUST persist every message (user + AI) to a relational database, associated with a session/conversation.
- **FR-004**: System MUST seed the LLM system prompt with store domain knowledge: shipping policy, return/refund policy, support hours.
- **FR-005**: System MUST validate that messages are non-empty and ≤ 2000 characters.
- **FR-006**: System MUST handle LLM API failures (timeouts, invalid key, rate limit) gracefully, returning a user-friendly fallback message instead of crashing.
- **FR-007**: System MUST support at least one real LLM provider (Gemini or OpenAI) configured via environment variables; API keys must never be hard-coded.
- **FR-008**: System MUST include last 10 messages of conversation history in the LLM prompt for contextual replies.
- **FR-009**: Frontend MUST render distinct chat bubbles for user (right-aligned, blue) and AI (left-aligned, dark) messages.
- **FR-010**: Frontend MUST auto-scroll to the latest message after each new turn.
- **FR-011**: Frontend MUST disable the send button and input while a request is in-flight.
- **FR-012**: Frontend MUST show an animated typing indicator while waiting for the AI response.
- **FR-013**: Frontend MUST persist `sessionId` in `localStorage` so conversation history survives page reloads.
- **FR-014**: Frontend MUST support sending a message via Enter key as well as the Send button.

### Key Entities

- **Conversation** (`chat_conversations`): Represents a chat session. Fields: `id` (UUID, PK), `created_at`, `updated_at`.
- **Message** (`chat_messages`): A single turn in a conversation. Fields: `id` (UUID, PK), `conversation_id` (FK → conversations, cascade delete), `sender` (`"user"` | `"ai"`), `text`, `timestamp`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can open `/chat`, send a question, and receive an AI-generated answer — end to end — without errors under normal conditions.
- **SC-002**: Conversation history survives a page reload: all previous messages are displayed in order.
- **SC-003**: The AI reliably answers questions about shipping, returns, and support hours using the seeded store knowledge.
- **SC-004**: No backend crash occurs when empty, oversized, or malformed payloads are sent.
- **SC-005**: When the LLM is unavailable or times out, the user sees a friendly error message rather than a blank screen or unhandled exception.
- **SC-006**: API keys are sourced exclusively from environment variables and are absent from source code.

---

## Assumptions

- The server environment provides `GEMINI_API_KEY` or `OPENAI_API_KEY` (or both) for real LLM replies; if neither is set the agent runs in demo mode.
- The browser supports `localStorage` for session ID persistence.
- No user authentication is required — the chat operates on anonymous sessions.
- Real external integrations (WhatsApp, Facebook, Shopify) are explicitly out of scope.
- Rich media (images, file attachments) is out of scope for this version.
